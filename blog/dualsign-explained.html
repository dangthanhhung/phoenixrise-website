DualSign: Why Two Algorithms Are Better Than One

When NIST finalized its post-quantum signature standards in 2024, two algorithms emerged: ML-DSA (FIPS 204) based on module lattices, and SLH-DSA (FIPS 205) based on hash functions. Most implementers pick one. Phoenix uses both simultaneously.

This isn't redundancy for the sake of redundancy. It's a deliberate engineering decision with deep cryptographic reasoning.

Different Math, Different Risks

ML-DSA's security relies on the hardness of the Module Learning With Errors (MLWE) problem. This is a lattice-based construction — a relatively new area of cryptography that has shown tremendous promise but has a shorter track record than hash-based approaches.

SLH-DSA's security relies only on the security of hash functions. Hash-based signatures have been studied since the 1970s (Lamport signatures). The mathematical assumptions are minimal — arguably the most conservative in all of cryptography.

These are fundamentally different mathematical foundations. A breakthrough that breaks lattice assumptions would not affect hash-based signatures. A weakness found in the specific hash construction would not affect lattice-based schemes.

The Cost of Being Wrong

In traditional software, you can patch a vulnerability and move on. In cryptography, being wrong is catastrophic. If the algorithm protecting a $100 million custody operation is broken, there is no patch — the signatures are already compromised.

DualSign eliminates single-algorithm risk. Both ML-DSA and SLH-DSA must verify for an attestation to be valid. An attacker must break both mathematical problems simultaneously — a dramatically higher bar than breaking either one alone.

Performance Tradeoffs

ML-DSA is fast: signing takes roughly 2 milliseconds with compact signatures around 3.3 KB. SLH-DSA is slower — signing takes roughly 50 milliseconds with larger signatures around 7.8 KB (for SLH-DSA-128s).

For Phoenix's use case — attestation of cryptocurrency transactions — this performance cost is negligible. Transaction settlement times are measured in seconds to minutes. An additional 50ms for a second quantum-safe signature is invisible.

The storage overhead (roughly 11 KB per dual attestation) is similarly trivial compared to the value being protected. When you're securing millions in digital assets, 11 KB of signature data is not the bottleneck.

When One Algorithm Falls

Cryptographic history is filled with algorithms that were considered secure until they weren't. MD5 was widely used for decades before collision attacks made it obsolete. SHA-1 followed a similar path. RSA key sizes have been revised upward repeatedly.

We don't know when — or if — ML-DSA or SLH-DSA will face a similar breakthrough. But we can architect systems that survive the possibility. DualSign is that architecture.

If ML-DSA is broken tomorrow, every Phoenix attestation is still protected by SLH-DSA. If SLH-DSA faces an issue, ML-DSA continues to provide security. The system degrades gracefully instead of failing catastrophically.

In cryptography, the cost of defense-in-depth is measured in milliseconds. The cost of single-algorithm failure is measured in billions.
