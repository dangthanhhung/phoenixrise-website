Layer 1

PQC Signing Engine

ML-DSA (FIPS 204)

Module-Lattice Digital Signature Algorithm. Security from the Module Learning With Errors (MLWE) problem. Fast signing (~2ms), compact signatures (~3.3 KB). Primary signer for performance-critical operations.

Lattice-based~2ms signing3.3 KB signatures

SLH-DSA (FIPS 205)

Stateless Hash-based Digital Signature Algorithm. Security from hash functions only — the most conservative cryptographic assumption known. Larger signatures (~7.8 KB) but minimal mathematical assumptions. Secondary signer for maximum assurance.

Hash-based~50ms signingConservative

DualSign + Merkle Commitment

Both algorithms sign simultaneously. Both must verify. If a breakthrough breaks lattice assumptions, hash-based signatures remain secure. All epoch public keys committed to a Merkle tree for O(log n) verification.

Layer 2

DyLWE Deterministic Core

Why Not Just RNG?

External randomness (OS RNG) cannot be audited, reproduced, or tested deterministically. It's vulnerable to rollback attacks. PhoenixSig needs a core that evolves forward-only, produces unpredictable but reproducible output, and never depends on external entropy for state progression.

Module Ring Operations

DyLWE operates on R_q = Z_q[X]/(X^N+1). Learning With Rounding provides deterministic noise through rounding operations — no sampling, no external randomness in the evolution step. State size: ~few KB.

Epoch Key Derivation

seed_epoch = HKDF(VaultKey ∥ sigma ∥ ctx). Every signing key is derived from hardware secret + evolving state + operation context. Deterministic, auditable, but unpredictable without VaultKey.

Forward-Only State

DeviceState = { sigma, epoch, counter, refresh_pending }. State only moves forward — sigma evolves with every signing operation. Monotonic counters and state hash chains enforce anti-rollback.

+

Sigma evolves via one-way DyLWE transformation after every sign. Computing sigma_{t+1} from sigma_t is efficient; reversing is infeasible. Forward security even without Phoenix Injection.

Layer 3

Phoenix Injection — Post-Compromise Security

No PCS without new entropy after compromise. DyLWE alone is not sufficient. Epoch keys alone are not sufficient. Phoenix Injection is mandatory.

— The PhoenixSig design invariant

VaultKey & TEE

Each device holds a VaultKey in a Trusted Execution Environment (Android Keystore / Secure Enclave). Non-exportable. All key derivation flows through VaultKey. This is the PCS invariant — violate it, and PCS dies.

+

Generated inside TEE during initialization, never leaves hardware. Input to every HKDF — all operations depend on hardware-attested entropy surviving full OS/kernel compromise.

Phoenix Refresh

VaultKey ← HKDF(VaultKey ∥ new_secret, "phoenix"). After refresh, every future seed changes completely. An attacker with full RAM + storage snapshot before refresh has zero knowledge of future keys. The math is simple. The implications are profound.

+

VaultKey ← HKDF(VaultKey || new_secret, "phoenix"). new_secret from hardware RNG inside TEE. After refresh, every downstream derivation changes completely and unpredictably.

The PCS Invariant (Rule #1)

Every hash, KDF, PRF, and state evolution MUST depend on the current VaultKey: ctx = H("PhoenixSig|ctx", epoch, counter, message, policy, vault.salt32(...)). This rule is non-negotiable.

Quarantine Protocol

On reboot, suspected compromise, or sync loss: quarantine activates (refresh_pending = true). No real payloads signed — only pings and dummy operations. System waits for hardware-confirmed entropy injection before resuming real operations.
